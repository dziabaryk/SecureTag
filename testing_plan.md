# ПЛАН ТЕСТИРОВАНИЯ ИНФОРМАЦИОННОЙ СИСТЕМЫ «SecureTag»

Настоящий документ описывает стратегию, объекты, критерии и детализированные сценарии тестирования для системы управления биометрическими трекерами SecureTag.

## 2.1. Объекты тестирования

Основные функциональные и нефункциональные модули системы, подлежащие проверке:

| Модуль | Описание | 
 | ----- | ----- | 
| **AUT/User** | Модуль аутентификации и управления профилем пользователя. | 
| **Tracker Management** | Функционал сопряжения, переименования, сброса настроек устройства. | 
| **Location Tracking** | Получение, обработка и отображение гео-данных в реальном времени. | 
| **Security & Biometrics** | Активация режима пропажи, блокировка устройства, валидация биометрических хешей. | 

## 2.2. Подход к тестированию

* **Уровень тестирования:** Будет применяться многоуровневое тестирование: Unit (проверка отдельных функций), Integration (проверка взаимодействия модулей) и System (проверка полного рабочего процесса).

* **Тип тестирования:**

  * **Функциональное:** Применяется подход **"Черного ящика"** для проверки всех требований Технического Задания (например, сопряжение, активация режима пропажи).

  * **Нефункциональное:** Проверка производительности (нагрузочное тестирование при высоком потоке гео-данных) и безопасности (тестирование на проникновение, проверка защищенности биометрических данных).

* **Автоматизация:** Приоритет отдается автоматизированным тестам для уровней Unit и Integration. Ручное тестирование будет применяться для проверки UI/UX и критически важных функций безопасности.

## 2.3. Критерии начала и окончания тестирования

### Критерии начала (Entry Criteria)

1. Все модули скомпилированы и развернуты в тестовом окружении.

2. Критические дефекты в предыдущих сборках (Smoke/Sanity) исправлены.

3. Все тестовые данные готовы и загружены в тестовую БД.

### Критерии окончания (Exit Criteria)

1. Выполнено 100% запланированных тестовых сценариев.

2. Уровень дефектов класса 'Critical' и 'High' равен 0.

3. Процент прохождения тест-кейсов (Pass Rate) выше 95%.

4. Все обнаруженные блокирующие дефекты исправлены и повторно протестированы (Regression Passed).

## 2.4. Перечень тестовых сценариев (Тест-кейсов)

Разработаны детализированные тест-кейсы для ключевых сценариев использования системы.

### Тест-кейс TC-01: Успешное сопряжение нового трекера (Pairing)

| Параметр | Значение | 
 | ----- | ----- | 
| **ID** | TC-01 | 
| **Наименование** | Проверка успешной регистрации и привязки нового устройства. | 
| **Предусловия** | Пользователь авторизован. SecureTag включен, находится в режиме сопряжения (мигает индикатор) и в радиусе Bluetooth-соединения. | 
| **Шаги** | 1\. В Mobile App нажать «Добавить SecureTag». 2. Выбрать устройство в списке найденных. 3. Приложить палец к сканеру SecureTag для записи биометрического хеша. 4. Подтвердить имя трекера. | 
| **Ожидаемый результат** | 1\. Устройство успешно привязывается к аккаунту (статус 201 Created). 2. На Главном экране появляется новый трекер с заданным именем и статусом «Активен». 3. Сервер успешно сохраняет публичный ключ устройства и идентификатор биометрического хеша. | 
| **Фактический результат** | \- **Статус:** Pass/Fail | 

### Тест-кейс TC-02: Активация режима пропажи (Lost Mode)

| Параметр | Значение | 
 | ----- | ----- | 
| **ID** | TC-02 | 
| **Наименование** | Проверка активации режима пропажи и блокировки устройства. | 
| **Предусловия** | Пользователь авторизован. Трекер ST-001 активен и находится на связи с сервером. | 
| **Шаги** | 1\. На Главном экране выбрать трекер ST-001. 2. Перейти в Настройки $\rightarrow$ Режим пропажи. 3. Активировать режим, ввести код подтверждения. 4. Проверить статус устройства в приложении и на физическом трекере. | 
| **Ожидаемый результат** | 1\. Трекер получает команду на блокировку. 2. В приложении статус ST-001 меняется на «Пропал (Lost)». 3. На трекере активируется блокировка (отключается сканер отпечатков, если приложен чужой палец). 4. В базу данных записывается событие активации Lost Mode. | 
| **Фактический результат** | \- **Статус:** Pass/Fail | 

### Тест-кейс TC-03: Проверка состояния скоропортящихся ресурсов

| Параметр | Значение | 
 | ----- | ----- | 
| **ID** | TC-03 | 
| **Наименование** | **Проверка жизненного цикла ресурса и его пригодности.** | 
| **Предусловия** | Доступ к объекту тестирования в его текущем состоянии. | 
| **Шаги** | 1\. Визуально оценить цвет ресурса (от зеленого до желтого). 2. Произвести тактильную оценку твердости/мягкости. 3. Проверить наличие внешних повреждений или дефектов. 4. Сделать вывод о его текущей стадии зрелости. | 
| **Ожидаемый результат** | Ресурс соответствует стандарту, установленному для стадии "Optimal Ripeness" (Оптимальная зрелость) и может быть использован для запланированных операций. | 
| **Фактический результат** | \- **Статус:** Pass/Fail | 

## 2.5. Требования к окружению

| Компонент | Требование | Назначение | 
 | ----- | ----- | ----- | 
| **Backend** | Docker-контейнер с Node.js (v18+) и PostgreSQL (v15+) | Серверная логика и база данных. | 
| **Mobile App** | Устройство Android  или iOS | Проверка UI, UX, и функционала Bluetooth. | 
| **Инструменты** | Postman/Insomnia (для тестирования API); Jest/Mocha (для Unit-тестирования). | Автоматизированная и ручная проверка API/кода. | 
| **Трекер** | Физический SecureTag (прошивка v1.2.0) или его симулятор. | Проверка end-to-end взаимодействия. |
